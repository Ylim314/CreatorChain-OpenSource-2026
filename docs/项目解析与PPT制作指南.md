## 一、项目一句话描述

**CreatorChain** 是一个**基于区块链的数字创作版权保护平台**，通过创新的**双重确权机制**，为AI生成、人工创作、混合创作等所有类型的数字内容提供永久、可信、可验证的版权保护。

### 核心卖点（供PPT封面/开篇使用）

- 🎨 **多元创作支持** - 首个同时支持AI创作、手工创作、混合创作的版权平台
- 🔐 **双重确权机制** - 创新的两步确权流程，记录过程+确认结果
- 🛡️ **隐私保护** - 零知识证明技术，验证真实性同时保护创作隐私
- 💾 **永久存储** - IPFS去中心化存储，内容永不丢失
- 🤖 **AI深度集成** - 集成智谱、OpenAI、豆包、DeepSeek、Kimi、Qwen等15+主流AI模型

## 二、项目背景与痛点

### 2.1 行业背景

随着AI技术的爆发式发展，2025年AI生成内容(AIGC)市场规模已达数千亿，预计2030年将突破万亿。然而，AI创作带来的版权问题日益严峻。

### 2.2 核心痛点（可做成问题→方案对照图）

| 痛点                       | 具体表现                               | 影响               |
| -------------------------- | -------------------------------------- | ------------------ |
| **版权归属不明确**   | AI创作、人机协作作品的版权归属存在争议 | 创作者权益无法保障 |
| **创作过程难以验证** | 传统版权登记无法证明创作的真实过程     | 版权纠纷难以裁定   |
| **AI创作确权困难**   | 现有法律和技术对AI创作版权保护存在空白 | AI创作者缺乏保护   |
| **收益分配不公平**   | 中间商抽取大量佣金，创作者收益微薄     | 打击创作积极性     |
| **存储安全隐患**     | 集中式存储存在数据丢失和篡改风险       | 作品可能永久丢失   |
| **跨平台确权难**     | 不同平台间版权信息无法互认             | 维权成本高昂       |

### 2.3 市场需求数据（可用于PPT图表）

- 全球数字内容创作者超过 **5000万**
- AI生成内容年增长率超过 **300%**
- 版权纠纷案件年均增长 **40%**
- 创作者因版权问题年损失超过 **100亿美元**

### 5.2 核心功能详解

#### 功能1：多元化创作支持

- **AI创作**：集成15+主流AI模型
  - **国产大模型**：智谱GLM-4.6、GLM-4-Air、GLM-4V-Plus、CogView-4、豆包、DeepSeek、Kimi、Qwen
  - **国际模型**：OpenAI DALL-E 3、Stable Diffusion XL、Midjourney v6
  - **文本生成**：支持长文本创作（最大150K tokens）
  - **图像生成**：支持多种风格和尺寸（1024x1024等）
- **手工创作**：支持上传已有数字作品（图片、文档、视频等）
- **混合创作**：AI辅助+人工修改的组合创作，贡献度自动评估

#### 功能2：双重确权机制 ⭐（核心创新点）

CreatorNFT合约实现完整的三阶段确权流程：

```text
第一次确权（创作过程记录）          第二次确权（最终作品确认）
        ↓                                  ↓
┌─────────────────────┐          ┌─────────────────────┐
│ recordCreationProcess│    →    │ confirmFinalCreation│
├─────────────────────┤          ├─────────────────────┤
│ • 创作过程IPFS哈希   │          │ • 最终作品IPFS哈希  │
│ • 内容哈希           │          │ • 最终内容哈希      │
│ • 标题/描述          │          │ • 贡献度评分(0-1000)│
│ • 创作类型(AI/手工)  │          │                     │
│ • 时间戳证明         │          │ • 确认时间戳        │
│ • 创作者地址         │          │ • 防重复验证       │
└─────────────────────┘          └─────────────────────┘
        ↓                                  ↓
   Stage 0: ProcessRecorded    Stage 1: FinalConfirmed
   (过程已记录)                  (最终已确认)
                                           ↓
                              ┌─────────────────────┐
                              │  mintCreationNFT   │
                              ├─────────────────────┤
                              │ • 元数据URI         │
                              │ • ERC-2981版税设置  │
                              │ • NFT铸造完成       │
                              │ • 转移追踪初始化    │
                              └─────────────────────┘
                                           ↓
                                  Stage 2: NFTMinted
                                  (NFT已铸造)
```

**技术细节**：
- 使用`contentHashToTokenId`映射防止内容重复注册
- 支持ERC-721可枚举，可查询用户所有NFT
- 版税默认5%，创作者可调整（最高20%）
- 转移历史自动追踪，记录转移次数

#### 功能3：零知识证明隐私保护

- **Schnorr签名算法**：证明拥有秘密数据而不泄露内容
- **承诺方案**：Hash(原始数据+随机盐) → 链上存储
- **选择性披露**：用户可选择公开的信息级别

#### 功能4：IPFS分布式存储

- **内容寻址**：基于内容哈希的永久链接
- **多重备份**：本地节点+Pinata企业服务
- **防篡改**：内容一旦上传无法修改

#### 功能5：智能合约自动化

- **自动确权**：无需人工干预的确权流程
- **自动分成**：授权收入按比例自动分配
- **透明可查**：所有交易链上可验证

#### 功能6：积分激励系统

- **积分获取**：
  - 创作作品：+100积分
  - 完成确权：+50积分
  - 收藏作品：+10积分
  - 评论互动：+15积分
- **积分消费**：购买内容、解锁功能、使用AI模型
- **合规设计**：使用积分而非代币，符合监管，零Gas费用

#### 功能7：前端功能模块（22+页面）

**核心页面**：
- **首页**：炫酷粒子效果展示，功能导航
- **AI创作工作室**：多模型选择、参数调节、实时预览
- **手工创作**：文件上传、元数据填写、一键确权
- **我的作品**：作品管理、确权状态、NFT展示
- **我的收藏**：收藏作品列表、快速访问
- **我的授权**：已购买授权管理
- **市场**：作品浏览、筛选、搜索、交易
- **作品详情**：完整信息展示、区块链验证、授权购买
- **个人中心**：用户资料、积分余额、创作统计
- **治理投票**：DAO提案、投票参与、提案创建
- **数据洞察**：平台数据可视化、趋势分析
- **智能合约**：合约信息、交互历史、状态查询
- **区块链验证**：链上数据查询、真实性验证
- **社区生态**：社区动态、创作者展示
- **AI模型配置**：模型参数设置、API配置

### 7.2 主要智能合约

#### 合约1: SimpleCreationRegistry - 本地演示主合约

- **定位**：与前端 `blockchainService` 直接联调的主合约，用于本地演示和课程答辩。  
- **核心功能**：实现双重确权的最小版本，只关注“创作登记 + 最终确认”，不直接负责 NFT 铸造。  
- **关键数据结构**：`Creation` 结构体包含 id、creator、title、description、ipfsHash、creationType、contentHash、confirmed、timestamp 等字段。  
- **关键方法**：
  - `registerCreation()` - 第一次确权：登记创作过程的关键信息和内容哈希；
  - `confirmCreation()` - 第二次确权：确认最终作品的 IPFS 哈希和内容哈希，并将 `confirmed` 标记为 true。  
- **调用示意**：
  - 前端 `ManualCreation.handleSubmit` → `blockchainService.registerCreation` → `SimpleCreationRegistry.registerCreation`；  
  - 前端上传最终元数据后 → `blockchainService.confirmCreation` → `SimpleCreationRegistry.confirmCreation`。

#### 合约2: CreatorNFT - 版权NFT（扩展设计）

- **功能**：ERC-721 + ERC-2981版税标准 + ERC-721Enumerable
- **核心**：在 SimpleCreationRegistry 的确权结果基础上，扩展出 NFT 铸造和版税机制
- **关键方法**：
  - `recordCreationProcess()` - 第一次确权（记录创作过程）
  - `confirmFinalCreation()` - 第二次确权（确认最终作品）
  - `mintCreationNFT()` - 第三步铸造NFT
  - `updateRoyalty()` - 更新版税（最高20%）
  - `verifyCreation()` - 验证创作真实性
  - `getCreationInfo()` - 获取完整创作信息
- **安全特性**：
  - ReentrancyGuard防重入攻击
  - AccessControl基于角色的权限控制
  - 内容哈希唯一性检查
  - 转移历史追踪

#### 合约3: LicenseManager - 设计原型（不实际部署）

- **重要说明**：此合约为设计原型，不一定部署在本地演示环境
- **实际实现**：授权功能通过后端API实现，使用链下积分系统
- **授权类型**：
  - Personal (个人使用)
  - Commercial (商业使用)
  - Exclusive (独家授权)
  - OpenSource (开源授权)
- **支付方式**：使用链下积分，不涉及任何虚拟货币

#### 合约4: CreatorDAO - 设计原型（不实际部署）

- **重要说明**：此合约为设计原型，不一定部署在本地演示环境
- **实际实现**：治理功能通过后端API实现，使用链下积分系统
- **提案生命周期**：Pending → Active → Succeeded → Queued → Executed
- **投票权重**：基于积分余额，不涉及代币

#### 合约5: CreatorToken - 设计原型（不实际部署）

- **重要说明**：此合约仅为设计原型，实际积分由链下数据库实现。  
- **实际实现**：采用链下积分系统，存储在数据库中；积分用于平台内激励，不涉及任何虚拟货币。  
- **用途构想**：未来可以将链下积分平滑迁移到链上，用于激励、支付、治理等场景。

### 9.2 关键安全措施

#### 智能合约安全
- **OpenZeppelin安全库**：使用经过审计的标准库
- **ReentrancyGuard**：防止重入攻击
- **AccessControl**：基于角色的权限管理（MINTER_ROLE、VERIFIER_ROLE等）
- **内容哈希验证**：防止重复注册和内容篡改
- **版税上限**：限制版税不超过20%，防止恶意设置

#### 后端安全
- **签名验证**：Web3签名验证确保请求来源
- **时间戳防重放**：防止重放攻击
- **输入清理**：SQL注入防护、XSS防护
- **CORS配置**：SecureCORSMiddleware限制跨域访问
- **请求超时**：RequestTimeoutMiddleware防止长时间阻塞
- **API限流**：防止恶意请求和DDoS攻击
- **JWT认证**：用户身份验证和授权

#### 数据安全
- **零知识证明**：Schnorr签名+SHA256承诺方案保护创作隐私
- **HTTPS加密传输**：所有API通信加密
- **IPFS内容寻址**：基于哈希的内容完整性验证
- **数据库加密**：敏感数据加密存储

#### 前端安全
- **Web3安全**：MetaMask集成，私钥不离开用户设备
- **输入验证**：客户端和服务端双重验证
- **XSS防护**：React自动转义，防止脚本注入

## 十三、常见问题预案

### Q1: 为什么需要双重确权？一次确权不够吗？

**答**：传统确权只记录最终作品，无法证明创作过程。双重确权先记录创作过程（AI模型、提示词、参数等），再确认最终作品，形成完整的创作证据链，在版权纠纷时可以证明创作的真实性和时间优先。

### Q2: 零知识证明具体是怎么实现的？

**答**：我们采用Schnorr签名+SHA256承诺方案。创作时，系统将秘密数据（提示词、参数等）哈希后上链，原始数据保存在本地。验证时，用户提供原始数据，系统重新计算哈希对比验证。这样既能证明拥有秘密数据，又不泄露具体内容。

### Q3: AI创作的版权到底归谁？

**答**：我们的贡献度评估算法量化了人类在AI创作中的贡献（提示词设计30%、参数优化25%、迭代调整20%等）。高贡献度说明人类创作者投入了大量创意劳动，版权归属更倾向于创作者。

### Q4: 为什么用积分而不用代币？

**答**：考虑到监管合规性，我们采用积分制度。积分可以在平台内使用，但不涉及金融交易，符合相关法律法规，同时也避免了区块链交易的高额Gas费用。

### Q5: IPFS存储会不会丢失数据？

**答**：我们采用多层存储策略：本地IPFS节点+Pinata企业级Pinning服务+备用节点。Pinata保证数据永久保存，即使本地节点故障也不会丢失。

### Q6: 智能合约安全性如何保证？

**答**：1) 使用OpenZeppelin审计过的安全库；2) 实现重入攻击防护(ReentrancyGuard)；3) 基于角色的访问控制；4) 时间锁机制防止恶意操作；5) 紧急暂停功能应对突发情况。

### Q7: 这个项目和现有的NFT平台有什么区别？

**答**：1) 现有NFT平台只关注结果，我们关注过程+结果；2) 我们专门为AI创作设计了贡献度评估；3) 零知识证明保护创作隐私；4) 双重确权机制是业界首创。

### Q8: 如何部署和运行项目？

**答**：项目提供一键启动脚本（`启动项目.bat`），支持Windows环境。后端使用Go + Gin，前端使用React，智能合约使用Hardhat。详细步骤：
1. 配置环境变量（`.env`文件）
2. 启动本地区块链（Ganache/Hardhat Network）
3. 部署智能合约（`npx hardhat run scripts/deploy-enhanced.js`）
4. 启动后端服务（`go run cmd/api/main.go`）
5. 启动前端服务（`npm start`）

### Q9: 支持哪些区块链网络？

**答**：目前支持以太坊主网、Polygon、Sepolia测试网、本地Hardhat网络。可通过配置文件切换网络，合约地址自动适配。

### Q10: AI模型调用失败怎么办？

**答**：系统支持Mock模式，当API Key未配置时自动切换到演示模式。配置API Key后即可使用真实AI服务。支持15+主流AI模型，包括国产大模型（智谱、豆包、DeepSeek等）。

### Q11: IPFS上传速度慢怎么办？

**答**：系统采用Pinata企业级服务作为主要网关，上传速度稳定。同时支持本地IPFS节点作为备选。大文件建议使用Pinata，小文件可使用本地节点。

### Q12: 如何验证作品在区块链上的真实性？

**答**：通过作品详情页的"区块链验证"功能，输入作品哈希或Token ID，系统会查询链上数据并对比验证。所有确权记录永久保存在区块链上，可通过区块链浏览器查看。

### API接口说明

**主要API端点**：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/creations` | GET | 获取作品列表 |
| `/api/creations` | POST | 创建新作品 |
| `/api/creations/:id` | GET | 获取作品详情 |
| `/api/creations/:id/confirm` | POST | 确认最终作品 |
| `/api/ai/generate` | POST | AI生成内容 |
| `/api/ipfs/upload` | POST | 上传到IPFS |
| `/api/users/:address` | GET | 获取用户信息 |
| `/api/points/balance` | GET | 获取积分余额 |
| `/api/marketplace/listings` | GET | 获取市场列表 |
| `/health` | GET | 健康检查 |
