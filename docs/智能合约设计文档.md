# CreatorChain 智能合约设计文档

## 📋 概述

CreatorChain 是一个基于区块链的数字创作版权保护平台，通过智能合约实现去中心化的版权确权。本项目采用链下积分系统，不涉及任何虚拟货币。

## 📜 合约详解

### 1. CreatorToken - 设计原型（不实际部署）

**文件（当前实现路径）**: `contracts/contracts/CreatorToken.sol`

**重要说明**:
- 此合约仅为设计原型，不实际部署到任何区块链网络
- 本项目采用链下积分系统，存储在数据库中
- 积分仅用于平台内激励机制，不涉及任何虚拟货币
- 积分不能兑换成法币或任何形式的货币

**实际实现**:
- 积分系统：通过后端API实现，存储在PostgreSQL/SQLite数据库
- 积分获取：通过系统奖励（创作、确权、互动等）
- 积分使用：购买作品授权、用户间转账
- 积分管理：通过后端API，不涉及区块链

### 2. CreatorNFT - 版权NFT（扩展设计）

**文件（当前实现路径）**: `contracts/contracts/CreatorNFT.sol`

> 提示：在当前版本中，本地前端演示主要使用 SimpleCreationRegistry 进行创作登记与双重确权，CreatorNFT 用于展示“如何在确权基础上进一步铸造 NFT、增加版税和生命周期管理”等扩展能力。

**核心功能**:
- **双重确权机制** ⭐ 项目亮点
  1. 第一次确权：记录创作过程
  2. 第二次确权：确认最终作品
- ERC-721 + ERC-2981 版税标准
- 防伪溯源验证
- 转移历史追踪

**双重确权流程**:
```
创作开始                     创作完成                    NFT铸造
   │                           │                          │
   ▼                           ▼                          ▼
┌─────────────┐          ┌─────────────┐          ┌─────────────┐
│recordCreation│  ───►   │confirmFinal │  ───►    │ mintNFT     │
│   Process   │          │  Creation   │          │             │
├─────────────┤          ├─────────────┤          ├─────────────┤
│• 过程IPFS哈希│          │• 最终IPFS哈希│          │• 元数据URI  │
│• 内容哈希    │          │• 内容哈希    │          │• 版税设置   │
│• 标题/描述   │          │• 贡献度评分  │          │             │
│• 创作类型    │          │              │          │             │
│• 时间戳      │          │• 时间戳      │          │             │
└─────────────┘          └─────────────┘          └─────────────┘
      │                        │                        │
      ▼                        ▼                        ▼
  Stage: 0               Stage: 1                 Stage: 2
  ProcessRecorded        FinalConfirmed           NFTMinted
```

**关键代码**:
```solidity
// 第一次确权
function recordCreationProcess(
    string memory processIpfsHash,
    bytes32 processContentHash,
    string memory title,
    string memory description,
    uint256 creationType  // 0=AI创作, 1=手工创作
) external returns (uint256) {
    // 检查内容唯一性，防止抄袭
    require(contentHashToTokenId[processContentHash] == 0, "Content already registered");
    
    // 创建记录
    info.processIpfsHash = processIpfsHash;
    info.processContentHash = processContentHash;
    info.processTimestamp = block.timestamp;  // 时间戳证明
    info.stage = CreationStage.ProcessRecorded;
    
    emit ProcessRecorded(tokenId, msg.sender, processIpfsHash, processContentHash, block.timestamp);
    return tokenId;
}

// 第二次确权
function confirmFinalCreation(
    uint256 tokenId,
    string memory finalIpfsHash,
    bytes32 finalContentHash,
    uint256 contributionScore  // 创作者贡献度 0-1000
) external {
    require(info.originalCreator == msg.sender, "Not the creator");
    require(info.stage == CreationStage.ProcessRecorded, "Invalid stage");
    
    info.finalIpfsHash = finalIpfsHash;
    info.finalContentHash = finalContentHash;
    info.contributionScore = contributionScore;
    info.confirmTimestamp = block.timestamp;
    info.stage = CreationStage.FinalConfirmed;
    
    emit FinalConfirmed(tokenId, msg.sender, finalIpfsHash, finalContentHash, contributionScore, block.timestamp);
}
```

### 3. SimpleCreationRegistry - 简化创作登记合约（前端联调主合约）

**文件（当前实现路径）**: `contracts/contracts/SimpleCreationRegistry.sol`

**设计目的**:
- 为前端本地演示提供一份“足够轻量”的确权合约；
- 只关注“创作登记 + 最终确认”两个步骤，不涉及 NFT 铸造与复杂授权；
- 接口简单，便于在课程设计答辩时从前端一路跟到链上。

**数据结构（简要）**:
- `Creation` 结构体包括：
  - `id`：作品自增 ID；
  - `creator`：创作者地址（`msg.sender`）；
  - `title` / `description`：标题与描述；
  - `ipfsHash`：作品文件或元数据哈希/URL；
  - `creationType`：作品类型（图像/文本/音频等）；
  - `contentHash`：对核心信息做 keccak256 的哈希；
  - `confirmed`：是否完成第二次确权；
  - `timestamp`：登记时间。

**双重确权接口**:
- `registerCreation(...)`：
  - 第一次确权，登记作品信息和内容哈希；
  - 返回 `creationId`，前端/后端据此建立数据库记录。
- `confirmCreation(...)`：
  - 第二次确权，更新最终 IPFS 哈希和内容哈希；
  - 仅允许最初登记的创作者调用，将 `confirmed` 置为 `true`。

**事件**:
- `CreationRegistered`：记录首次登记时的关键信息；
- `CreationConfirmed`：记录最终确认时的最终 IPFS 哈希。

在你当前的 CreatorChain 项目中：
- `SimpleCreationRegistry` 是“演示用主角”，对应前端 `blockchainService.registerCreation / confirmCreation`；
- `CreatorNFT`、`LicenseManager`、`CreatorDAO` 等是“进阶设计”，展示未来可以扩展到 NFT、授权和治理。

### 4. LicenseManager - 版权授权管理

**文件（当前实现路径）**: `contracts/contracts/LicenseManager.sol`

**核心功能**:
- 多种授权类型（个人/商业/独家/开源）
- 时间限制授权
- 自动分成机制
- 授权转让

**授权类型**:
| 类型 | 价格倍数 | 可转让 | 使用限制 |
|------|----------|--------|----------|
| Personal | 1x | 是 | 100次 |
| Commercial | 5x | 是 | 无限 |
| Exclusive | 20x | 否 | 无限 |
| OpenSource | 0.5x | 是 | 无限 |

**分成机制**:
```solidity
function _processPayment(uint256 tokenId, uint256 amount) internal {
    // 平台费 2.5%
    uint256 platformFee = (amount * platformFeeBps) / 10000;
    uint256 creatorAmount = amount - platformFee;
    
    // 按配置分成给创作者/贡献者
    for (uint256 i = 0; i < share.recipients.length; i++) {
        uint256 recipientAmount = (creatorAmount * share.shares[i]) / 10000;
        paymentToken.transferFrom(msg.sender, share.recipients[i], recipientAmount);
    }
}
```

**提案生命周期**:
```
Pending ─► Active ─► Succeeded ─► Queued ─► Executed
   │          │          │

   │          │
   │          └── 取消 ──► Canceled
   │
   └── 投票延迟期
```

**治理参数**:
| 参数 | 默认值 | 说明 |
|------|--------|------|
| votingDelay | 1天 | 提案创建后多久开始投票 |
| votingPeriod | 7天 | 投票持续时间 |
| executionDelay | 2天 | 通过后多久可执行 |
| proposalThreshold | 10000 积分 | 提案创建门槛（设计参考） |
| quorumVotes | 100000 积分 | 法定投票人数（设计参考） |
| passRate | 50% | 通过所需赞成比例 |

## 📊 技术亮点（期末答辩要点）

### 1. 双重确权创新
- **问题**: 传统版权登记无法证明创作过程
- **解决**: 两阶段上链，记录创作过程+最终成果

### 2. 链下积分系统
- **机制**: 积分存储在数据库中，通过后端API管理
- **合规**: 积分仅用于平台内激励，不涉及任何虚拟货币
- **功能**: 创作奖励、授权购买、用户转账

### 3. 去中心化确权
- **区块链存证**: 版权信息永久存储在区块链上
- **不可篡改**: 确权记录无法被修改或删除
- **透明可验证**: 任何人都可以验证版权归属

### 4. IPFS分布式存储
- **内容存储**: 作品内容存储在IPFS网络
- **永久保存**: 内容哈希存储在链上，确保永久可访问
- **去中心化**: 不依赖单一服务器
